# Playbook

## What to Capture From the Browser

Capture these before trusting any browser-vs-local comparison:

- Page URL
- Actual worker URL
- Session/media ID used by the browser worker
- Actual playlist URL and actual segment URLs
- Main-worker `postMessage` sequence and order
- State sequence generated by the worker after each update step

When possible, capture the browser main worker only. Audio-worker traffic often adds noise.

## Minimal Browser Instrumentation

Patch `window.Worker` early so you can:

- record `init`
- record the worker URL actually created by the page
- record all `postMessage` payloads sent to the main worker
- optionally patch worker internals to log state transitions

When patching a worker, prefer:

1. wrap `Worker`
2. identify the main worker by its `init.options.loaderType`
3. keep logs as small JSON rows so they survive long sessions

## Local Reproduction Order

Reproduce in this order:

1. Use the same session/media ID
2. Use the same fixed time if the browser session is time-sensitive
3. Use the same page URL and worker URL shape
4. Use the same actual segments, not guessed variants
5. Replay the same main-worker message order
6. Only after that, reduce to a lower-level runtime or single-unit experiment

## Zero-Output Analysis

When a worker call returns zero bytes:

1. Log the call ordinal
2. Log the timing tag
3. Log the current state tag
4. Map the call back to the raw media unit
5. Compare the previous and next calls

At minimum, keep:

- segment index
- call ordinal
- timing tag
- state tag
- input length
- output length
- media-unit type
- raw timing fields such as DTS/PTS when available

## Reduction Strategy

Use these experiments in order:

1. Neighbor-window replay
   Compare `ordinal - 1`, `ordinal`, `ordinal + 1`.

2. Alternate timing-tag replay
   Try nearby timing values on the same unit while holding the session constant.

3. Skip-ordinal replay
   Skip an earlier unit to see whether the later failure still happens.

4. Browser-message replay
   Replay the browser main-worker action order locally.

5. State-window capture
   Snapshot worker state before and after the relevant call and compare changed words.

## Interpreting Divergence

Common outcomes:

- Same input, different state sequence:
  The bug is in session buildup, not in one isolated media unit.

- Same state sequence, different output:
  The bug is in environment, lower-level runtime behavior, or binary differences.

- Zero output only on one path:
  Compare the earliest state divergence before the failing ordinal.

- Skipping one ordinal delays but does not remove the failure:
  More than one earlier unit contributes to building the bad state line.

## Good Output Artifacts

Keep compact artifacts that are easy to diff:

- `browser-main-sequence.json`
- `worker-zero-map.json`
- `state-window-ordinal-N.json`
- `skip-ordinal-N.json`
- `tag-variant-ordinal-N.json`

Each artifact should state:

- what input set was used
- which session/media ID was used
- which fixed time was used
- which ordinals or tags were targeted
